VMware Fusion 系列，理解 NAT 网络通信

中 EN

这是一篇写于当时语境下的技术记录。技术发展日新月异，若其中某些方法、结论或版本信息已经过时，就把它当作一次技术考古吧。

硬件：Apple M4
系统：Mac OS Tahoe 26.3
VMware Fusion：Professional 26H1（25388279）
虚拟机 OS：Ubuntu 26.04 LTS

如今 VMware Fusion 对于个人用户已经完全免费了，这可真是个天大的利好。无论是工作搬砖还是学习实验，经常需要本地虚拟化环境支持，而 VMware Fusion 绝对是个人开发者的首选平台。所以，接下来我会通过几篇小文章记录一下利用 VMware Fusion 在本地搭建实验环境时的一些配置方法和个人理解。

第一篇的话先做些知识铺垫，理解 NAT 模式下的网络通信原理。

VMware Fusion NAT 模式下的三个关键服务

在 NAT 模式（VMware Fusion 中对应 Share with my Mac）和 Host-Only 模式（VMware Fusion 中对应 Private to my Mac）下，VMware Fusion 通过内部核心服务协同工作，构建了一个虚拟网络环境，让虚拟机可以顺畅地访问外部网络以及与宿主机之间互访。

启动 VMware Fusion 后，我们通过以下命令输出来验证这一点并梳理清楚他们之间的关系：

# 首先查看下 VMware Fusion 的网络配置
sudo cat /Library/Preferences/VMware\ Fusion/networking

上图输出可以看到，VMware Fusion 会根据配置创建出 VNET_1、VNET_8 两套虚拟网络环境（分别对应输出中红框和绿框部分），它们各自内部开启 DHCP 服务，并分配了不同的私有网段。其中绿色框的 VNET_8 中还开启了 NAT 服务，因此它对应 NAT 模式；而红框的 VNET_1 只开启 DHCP，它对应 Host-Only 模式。

# 查看 VMware Fusion 网络服务状态
sudo /Applications/VMware\ Fusion.app/Contents/vmnet-cli --status

上图输出再次证明了 VMware Fusion 网络由多个内部服务协同工作的说法。

# 从系统进程层面查看
ps -ax -o pid,args | grep "vmnet-" | grep -v "grep"

上图输出可以看到名为 vmnet-bridge、vmnet-nat 和两个 vmnet-dhcpd（使用不同配置文件启动，分别服务 VNET_1、VNET_8） 的进程在工作。

# 从宿主机角度查看 VMware Fusion 对当前网络环境的影响
ifconfig

输出中红框的 bridge100 和 绿框的 bridge101 可以理解为是由 vmnet-bridge 这个进程分别为 VNET_1、VNET_8 两个虚拟网络环境创建的虚拟交换机，vmenet0、vmenet1、vmenet2 是交换机上的接口，创建虚拟机后，虚拟机相当于直连在这个交换机的接口上。

综上信息，得出结论，在 NAT 模式下工作时（VNET_8 虚拟网络环境）：

  • vmnet-bridge：理解为二层交换机，连接所有虚拟机和宿主机。
  • vmnet-dhcpd：服务于私有（虚拟）网络中的 DHCP 服务。
  • vmnet-natd：可理解为路由器（NAT 网关），虚拟机访问外部资源都通过它。

虚拟机与宿主机互通

# 重点关注这个对应 VNET_8 虚拟网络环境的虚拟网桥接口
ifconfig bridge101

输出中红框部分显示了这个虚拟网桥接口 IP 地址为 172.16.29.1。这个地址将作为宿主机在 VNET_8 这个虚拟网络中使用的 IP 地址。

# 查看服务于 VNET_8 虚拟网络环境的 DHCP 服务配置
sudo cat /Library/Preferences/VMware\ Fusion/vmnet8/dhcpd.conf

DHCP 配置红框部分显示了一个静态绑定的地址分配规则，正是上面 bridge101 网桥接口的地址。

# 在宿主机上启动一个 http 服务，用于验证虚拟机对宿主机（172.16.29.1）的访问。
python3 -m http.server 8000
# 从虚拟机上测试与宿主机网络连通性
curl -I http://172.16.29.1:8000/

通过访问 172.16.29.1 地址的 8000 端口成功，说明与宿主机通过 172.16.29.1 地址通信正常。

# 从虚拟机上测试与宿主机网络连通路径
traceroute 172.16.29.1

根据路由跟踪信息，访问 172.16.29.1（宿主机），通过虚拟网桥同网段二层直连，不经任何网关。

虚拟机与虚拟机互通

原理与 虚拟机与宿主机互通 一样，都是透过 虚拟网桥同网段二层直连，不经任何网关。

虚拟机与外部资源互通

# 查看服务于 VNET_8 虚拟网络环境的 DHCP 服务配置
sudo cat /Library/Preferences/VMware\ Fusion/vmnet8/dhcpd.conf

DHCP 配置绿框部分设置了虚拟机将被分配的默认网关地址为 172.16.29.2，这是 vmnet-natd 服务（相当于虚拟网络中的路由器）的地址。

# 在虚拟机上查看默认路由
ip route

默认路由（网关设置）指向了 172.16.29.2 这个地址，也就是说虚拟机所有对外部资源的访问都将透过 vmnet-natd 进行 NAT（地址转换）。

# 从虚拟机上测试下对外部资源的访问
traceroute www.baidu.com

红框部分显示了虚拟机访问外部资源路径的第一条就是 172.16.29.2。

总结

以上就是 VMware Fusion 网络工作在 NAT 模式下，虚拟机内部互通以及对外部资源访问的大致梳理。下一篇笔记会记录一下在此基础上配置虚拟机通过宿主机上的梯子代理访问外部资源，一劳永逸解决实验中依赖外网镜像仓库、Github 拉取代码等现实问题。
