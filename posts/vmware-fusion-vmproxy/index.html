VMware Fusion 系列，虚拟机配置代理访问外部资源

2026-07-10 虚拟化 #工具 #VMware Fusion #网络
中 EN

硬件：Apple M4
系统：Mac OS Tahoe 26.3
VMware Fusion：Professional 26H1（25388279）
虚拟机 OS：Ubuntu 26.04 LTS

“墙” 内的网络环境对于开发者不太友好，为了顺利访问各种官方源/仓库，“梯子” 必不可少，省去了各种国内源替换配置。那么 VMware Fusion 管理的虚拟机如何搭上宿主机上的这把 “梯子” 呢？

VMware Fusion 虚拟机配置 http 代理

上一篇文章，大致理清了 VMware Fusion 网络运行原理，下面在此基础上配置下虚拟机的 http 代理。

宿主机代理软件

宿主机上代理软件开启允许局域网访问，记住端口 7897。

VMware Fusion 虚拟机

配置代理前无法顺利访问 Google。

# 设置 http 代理
export http_proxy=http://172.16.29.1:7897

# 设置 https 代理
export https_proxy=http://172.16.29.1:7897

# 再次测试访问 Google，-v 参数输出调试信息
# 这里需注意，grep 只处理 stdout 标准输出
# 流，而调试数据走 stderr 标准错误，所以需
# 要 2>&1 先进行合流。
curl -I https://www.google.com -v 2>&1 | grep -E "200|proxy"

curl、wget、apt、pip 等命令的标准实现中，会读取 http_proxy、https_proxy 环境变量。可以看到截图中访问 Google 成功，并且调试信息部分（红框）明确指出了是通过环境变量设置的代理服务进行的访问。

# 测试 docker 拉取镜像
docker run --rm hello-world

继续测试 docker 拉取，发现仍然没有成功，这说明 docker 服务启动时不会读取 http_proxy、https_proxy 环境变量，使用代理需要单独配置。

sudo vim /etc/docker/daemon.json

{
  "proxies": {
    "http-proxy": "http://172.16.29.1:7897",
    "https-proxy": "http://172.16.29.1:7897",
    "no-proxy": "localhost,127.0.0.1,.local"
  }
}

sudo systemctl restart docker

再次尝试 docker 拉取，成功。

git 配置代理

git 命令虽然读取系统环境变量，但其对大小写敏感，需设置：

export HTTP_PROXY=http://172.16.29.1:7897
export HTTPS_PROXY=http://172.16.29.1:7897

或者

# 此方法设置后，优先级优于系统环境变量。
git config --global http.proxy http://172.16.29.1:7897
git config --global https.proxy http://172.16.29.1:7897

总结

至此 VMware Fusion 虚拟机配置代理访问外部资源成功。
