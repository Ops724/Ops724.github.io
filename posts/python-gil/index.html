GIL Python 全局解释器锁

2026-07-23 编程语言 #python
中 EN

这是一篇写于当时语境下的技术记录。技术发展日新月异，若其中某些方法、结论或版本信息已经过时，就把它当作一次技术考古吧。

GIL（全局解释器锁），它不是你写代码时自己加的锁（线程模块里实现的锁），而是 CPython（python 的主流实现）中的一个全局互斥锁，是解释器天生就有的，核心功能是确保同一时刻只有一个线程能执行 python 字节码。这牺牲了语言的并发能力，限制了多线程编程对多核 cpu 的利用。这也是常听有人攻击 python 性能差、伪多线程的原因，那为什么 CPython 还要搞 GIL 呢？

两点，历史原因 + 实现简单：

  • 早期 python 作者为了快速实现解释器
  • 为了让 C 扩展模块、内存管理、垃圾回收变得简单

所以说 GIL 是 CPython 解释器的实现选择，不是多线程必需的东西（下面深入讨论）。

Python 多线程完全没用？

cpu 密集型任务

对于 cpu 密集型任务，python 多线程确实没有提升，甚至会令处理性能更差。因为 GIL 的存在，对于一个 cpu 密集型任务，多线程相比单线程串行处理在整个工作量不变的情况下，多出了线程间频繁争抢 GIL 的切换开销。

io 密集型任务

对于 io 密集型任务，python 多线程仍然高效。高效的原因并不是说不受 GIL 锁影响，GIL 锁是解释器级别的，它一直存在，仍然是同一时刻仅允许一个 python 线程执行字节码，但对于 io 密集型任务来说，当线程触及到 io 操作（等待网络、文件读写、sleep 等）时，GIL 锁会被主动释放，下一个正在等待的线程会立刻获得开始任务，本质是这大大提升了 cpu 的利用率，一定比单线程串行效率更高。

为什么说 GIL 简化了垃圾回收实现？

核心原因是，引用计数（python 使用的 GC 策略）的线程安全问题。python 中每个对象底层都有一个字段记录着有多少个引用指向它，如果没有 GIL 就会出现严重的竞态条件，假设两个线程同时操作一个对象：

  • 线程A：读取引用计数（假设当前为 1）
  • 线程B：读取引用计数（也是 1）
  • 线程A：计数增加到 2
  • 线程B：计数增加到 2（实际应该是 3）

结果：会导致对象过早的释放。

上述问题的解决办法：

方案1：像其它语言那样，使用细粒度的锁（如果不使用 GIL，每个对象都需要加锁，这里指的是解释器级别，不是应用程序中的锁）。

问题：

  • 性能开销巨大：python 中引用计数的操作极其频繁，如果每个对象都有锁
  • 内存开销：每个对象都要额外存储锁信息
  • 死锁风险：复杂对象的相互引用可能导致的死锁
  • 锁竞争：热门对象会被大量线程争抢

方案2：GIL（CPython 的选择）

优势：

  • 零额外内存开销
  • 零锁操作开销
  • 无死锁风险
  • 代码简洁

GIL 方案通过牺牲多核并发性能换取了这些。python 之父曾说到：GIL 是 CPython 的特性，而不是 bug，它给 CPython 带来了简单性和可维护性。

为什么其它编程语言不需要 GIL ？

首先确定一点，GIL 确实是 CPython 独有的特性，它是历史包袱，并不是多线程必需的。像 java、golang、c++ 等它们都没有 GIL ，所以这些语言可以做到真正的多线程，充分利用多核 cpu。因为这些语言用了更复杂、更精细的机制，比如细粒度的锁（锁对象）、严格的内存模型等，它们不锁整个解释器，只锁需要保护的小部分数据。

为什么 Python 有了 GIL 还需要线程锁的存在？

无论有无 GIL，只要涉及到多线程对共享数据的操作就需要线程锁（自己加锁），这是所有语言的通用规则。因为 GIL 的默认行为是同一时刻只有一个线程执行 python 字节码，执行一段字节码（python 3.2 后改为基于时间）就会让出 GIL 给别的线程再执行，所以说 GIL 切换会打断你的代码。举个例子：

counter += 1
# 这种代码在 python 底层运行逻辑中会被拆解为：
# 1. 读取 counter 值（GIL 可能在这里释放）
# 2. 计算 counter + 1（GIL 可能释放）
# 3. 写回 counter（GIL 可能释放）
# GIL 只保护原子性操作，上面这种复合操作需要自己加锁才行。

下表直观对比：

情况 python（有 GIL） go/java（无 GIL）
多线程能多核运行 不能 能
线程安全要不要手动加锁 要（GIL + 自己的锁） 要（自己的锁）

Jython 为什么是绕过 GIL 的手段？

因为 Jython 是把 python 代码翻译成 java 字节码在 jvm 虚拟机中执行，jvm 虚拟机没有 GIL 锁。

去掉 GIL 的可能性？

理论上完全可能

像其它编程语言一样，编程时不依赖 GIL 能力，更严格、更精细地自行处理线程，这通常意味着更细粒度的锁、原子操作（少加锁、加细锁、只锁数据、不锁逻辑）。

但需要注意的是，由于 GIL 的存在，python 程序员长久以来的习惯是并不在意加锁的范围（因为反正有 GIL 的存在永远都是锁线程），往往把整段逻辑和数据一同上锁，这会导致多核利用率下降，甚至性能不如单线程执行。

结论：良好的编程习惯 + 无 GIL = 真正的多核并行

GIL 的未来

python 3.13 引入了实验性的自由线程模式（–disable-gil 构建），允许在禁用 GIL 的情况下运行。但这只是可选功能，并不是默认行为。python 发展至今，大量生态是基于 GIL 约束进行的，所以完全移除 GIL 需要在不显著降低单线程性能的前提下解决大量兼容性问题，目前仍是一个长期演进的方向。
